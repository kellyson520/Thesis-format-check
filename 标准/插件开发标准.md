# 论文格式检查器 - 插件开发标准 (Rule Plugin Standard V1.0)

为了确保插件系统的可维护性、动态可配置性以及与前端 UI 的无缝对接，所有新增或重构的插件必须严格遵守本标准。

## 🎯 1. 核心架构与类定义

所有插件必须继承对应的核心接口，并混入 `DeclarativeConfigMixin` 声明配置能力。

- **段落级插件**: 继承 `BaseRulePlugin` 和 `DeclarativeConfigMixin`。
- **章节级插件**: 继承 `ISectionPlugin` 和 `DeclarativeConfigMixin`。

```python
class MyCoolPlugin(BaseRulePlugin, DeclarativeConfigMixin):
    plugin_id = "my_cool_feature"  # 1. 必须全局唯一，建议使用小写下划线
    config_model = MyCoolConfig    # 2. 必须指向对应的 Pydantic 模型
```

## 🏗️ 2. 配置模型规范 (Config Model)

每个插件必须拥有一个独立的 Pydantic 配置模型，且**必须**定义 `enabled` 字段。

```python
from pydantic import BaseModel

class MyCoolConfig(BaseModel):
    enabled: bool = True       # 核心开关，默认开启，由「系统中心」管理
    parameter_a: float = 1.0   # 业务参数
    # 其他业务参数...
```

## 🛠️ 3. 构造函数与依赖注入

插件必须通过构造函数接受 `RuleConfig` 注入，以便动态读取 `rules.yaml` 中的规则变更。

```python
def __init__(self, config: RuleConfig):
    self.config = config
```

## 🛡️ 4. 逻辑卫句规范 (The Guard Rule)

在执行任何计算逻辑前，**必须**检查开关状态。这是确保「系统设置」总控生效的唯一机制。

```python
def check(self, node: ParagraphNode, context: RuleContext) -> List[Issue]:
    issues: List[Issue] = []
    
    # 强制卫句：检查对应的 plugin_id 是否启用
    if not self.config.my_cool_feature.enabled:
        return issues
    
    # 执行业务逻辑...
    return issues
```

## ✅ 5. 注册与合规门禁 (Registration)

符合此标准的插件将自动出现在「系统设置」中心的模块总控列表中。集成路径包括：
1. **模型定义**: 在 `src/use_cases/rule_config.py` 中添加配置字段。
2. **流水线挂载**: 在 `src/use_cases/validator_pipeline.py` 中实例化插件。
3. **UI 适配**: 在 `src/main.py` 的 `plugin_display_names` 映射中增加显示名称。
