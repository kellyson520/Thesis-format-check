# 插件开发标准 (Rule Plugin Standard)

## 🎯 1. 插件定位与核心接口

在项目 `src/use_cases/plugins/` 目录下的所有类，统称为校验插件。它们必须遵守以下核心契约：

- **基类**: 必须继承 `src.domain.interfaces.BaseRulePlugin`。
- **唯一职责**: 一个插件仅负责核查文档的一个特定维度（如：字体、行距、页码、公式...）。
- **无副作用**: 插件必须是纯粹的逻辑计算，严禁在 `check()` 方法中修改外部状态或执行文件 I/O。

## 🏗️ 2. 核心架构红线 (Red Lines)

1.  **禁止耦合 UI/Infrastructure**: 插件内部严禁导入 `fastapi`, `frontend`, `infrastructure.parsers` 等。
2.  **强制声明式配置 (Declarative Config)**:
    - 插件必须继承 `use_cases.plugins.mixin.DeclarativeConfigMixin`。
    - 必须显式定义 `plugin_id` (对应 rules.yaml 中的键名)。
    - 必须定义 `config_model` (Pydantic 类)，声明其所需的所有参数。
3.  **零阻塞进度报告**: 利用 Python `generator` 机制逐段或逐条规则产出 `Issue` 列表。

## 🛠️ 3. 开发标准步骤 (SOP)

1. **接口对齐**: 确认 `domain/models` 的 `ParagraphNode` 或 `TableNode` 已包含该插件所需的元数据。
2. **逻辑实现**: 
   - 优先识别段落所在的 `Section` 类型（正文、摘要、参考文献等）。
   - 实现精确的校验算法，减少误报（False Positives）。
3. **输出标准化**: 返回 `src.domain.models.Issue` 对象，包含：
   - `message`: 简洁明确的错误提示（中、英文双语推荐）。
   - `severity`: 错误等级（Error, Warning, Info）。
   - `location`: 准确的定位元数据（Paragraph Index, Character Range）。
   - `suggestion`: (可选) 推荐的修复措施，供 `DocumentFixer` 使用。

## ✅ 4. 质量门禁 (Quality Gates)

- **单元测试**: 每个插件必须在 `tests/unit/use_cases/plugins/` 拥有对应的测试文件。
- **性能评估**: 对于 100 页以上的长文档，单个插件的平均解析/核查耗时不得超过 10ms/段落。
- **静态扫描**: 必须通过 `architecture-auditor` 对导入路径的纯净性校验。
