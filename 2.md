针对当前个人版 MVP 中存在的“上帝类（God Class）”聚集、业务逻辑与底层解析深度绑定等问题，要实现**职责单一化 (SRP)** 和**不反向依赖 (DIP)**，需要引入**领域驱动设计 (DDD) 的思想和整洁架构 (Clean Architecture)**。

以下是具体的架构改进和重构方案：

### 一、 职责单一化 (Single Responsibility)：拆解“上帝类”

目前 `DocxParser` 和 `Validator` 承担了太多本不属于它们的职责。我们需要将它们按照“纯逻辑”与“脏活（I/O与解析）”进行彻底剥离。

#### 1. 隔离 `DocxParser`：做纯粹的“翻译官”
* **现状痛点**：当前的 Parser 在解析时混入了业务逻辑（例如把 `pt` 硬除以 `12` 算字符数）。
* **改进方案**：`DocxParser` 不应该知道什么是“正确”，甚至不应该知道什么是“规则”。它的唯一职责是将 `python-docx` 极其复杂的 XML DOM 树，**翻译成系统自定义的“抽象文档模型 (Abstract Document Model, ADM)”**。
* **重构结果**：Parser 只输出纯粹的 Python Dataclass（如 `DocumentNode`, `ParagraphNode`, `TextRunNode`）。所有的单位换算（如 pt 转 cm、pt 转字符）应该交给专门的 `UnitConverter` 工具类处理。

#### 2. 瓦解 `Validator`：引入插件化流水线 (Pipeline & Plugin Pattern)
* **现状痛点**：`validator.py` 里的 `validate` 方法是一个巨大的 `for` 循环，包含了字体检查、正则匹配、层级追踪等所有逻辑，违反了开闭原则 (OCP)。
* **改进方案**：将 `Validator` 重构成一个**执行引擎 (Runner)**，具体的校验逻辑下放到各个独立的插件中。
* **代码形态示例**：
  定义一个基类 `BaseRulePlugin`：
  ```python
  class BaseRulePlugin(ABC):
      @abstractmethod
      def check(self, node: Node, context: Context) -> List[Issue]:
          pass
  ```
  然后拆分出：
  * `FontFamilyPlugin`：只管字体对不对。
  * `IndentPlugin`：只管缩进对不对。
  * `NumberingContinuityPlugin`：只管图表编号是否连续（自带内部状态机）。
  * `HierarchyPlugin`：只管标题是否跳级。
  这样，新增学校要求只需新增 Plugin 文件，核心引擎代码一行都不用改。

#### 3. 强化 `RuleLoader`：引入强类型配置 (Typed Models)
* **现状痛点**：目前规则是扁平的字典，Validator 中充斥着 `rules.get('headings', {}).get('level_1')` 这种脆弱的硬编码取值。
* **改进方案**：引入 `Pydantic`。`RuleLoader` 读取 YAML 后，必须将其反序列化为强类型的 Pydantic 模型（如 `class HeadingRule(BaseModel)`）。这能让配置在加载的瞬间完成自我校验（格式错误直接报错），Validator 中就能获得 IDE 的代码补全和类型安全。

### 二、 不反向依赖 (Dependency Inversion)：切断底层对上层的感知

“反向依赖”通常发生在**底层模块需要向上层（如 UI 或 API 控制器）报告进度或请求交互**时。为了彻底切断这种联系，必须运用**控制反转 (IoC)** 和**依赖注入 (DI)**。

#### 1. 解决进度上报的反向依赖：使用生成器 (Generator) 或事件发射器 (Event Emitter)
* **错误的做法**：`Validator` 在循环里调用 `fastapi_app.update_progress(50)`。这导致引擎依赖了 Web 框架。
* **正确的做法**：引擎层 (Engine) 应该是“盲”的。
  **方式 A (Yield)**：把 `validate` 变成一个生成器。
  ```python
  # validator_runner.py (底层引擎)
  def run_validation(self, doc_model, rules):
      total = len(doc_model.paragraphs)
      for i, para in enumerate(doc_model.paragraphs):
          issues = self._apply_plugins(para)
          # 不调用上层，而是产出当前状态
          yield {"progress": (i / total) * 100, "issues_found": issues} 
  ```
  **方式 B (Callback Injection)**：在调用引擎时，从上层把回调函数注入进去。
  ```python
  # main.py (顶层 FastAPI)
  def progress_callback(percent):
      websocket.send(f"progress:{percent}")

  engine.validate(doc, callback=progress_callback) # 引擎内部只管调 callback，不知道是谁在听
  ```

#### 2. 整洁架构分层 (Clean Architecture Layers)
重构后的项目结构应该形成严格的**单向依赖箭头**（外层依赖内层，内层对绝外层一无所知）：

* **Level 1: Domain Layer (内环 - 核心实体)**
  * *内容*：纯 Python 数据结构，如 `Issue` 模型、`DocumentNode` 抽象语法树、`RuleContext`。
  * *依赖*：什么都不依赖。
* **Level 2: Use Cases Layer (中环 - 业务用例)**
  * *内容*：`ValidatorPipeline` (调度插件), `DocxParser` (接口定义)。
  * *依赖*：仅依赖 Domain Layer。
* **Level 3: Infrastructure / Interface Layer (外环 - 基础设施)**
  * *内容*：`python-docx` 的具体调用逻辑、`PyYAML` 解析、`FastAPI` 路由、`PyWebView` 窗口控制、`Reporter` (生成批注 Word)。
  * *依赖*：依赖 Use Cases 和 Domain。负责把 HTTP 请求或文件流组装好，喂给中环的用例。

---

### 三、 引入“中介者”解耦上下文 (Context Decoupling)

前面提到过，图表编号、参考文献状态存在“泄露”的 Bug。这是因为各个段落之间强依赖了上下文状态。

* **改进方案**：引入一个 `DocumentContext` 对象作为中介者。
* 在解析文档时，先进行一次**预扫描 (Pre-pass)**，构建文档的骨架目录（如：第1页到第5页是摘要，第6页到第50页是正文，且正文包含3个一级标题）。
* 当 Plugin 在校验某个具体段落时，它不自己去维持 `in_references` 这种全局变量，而是向 `DocumentContext` 询问：“我当前这个节点处于论文的哪个区域？” Context 返回 `Enum(Section.REFERENCES)`，Plugin 据此决定应用什么规则。

### 总结：重构实施路径建议

如果要在接下来的开发中落地上述理念，建议按以下顺序开刀：
1. **先造防弹衣**：引入 `Pydantic` 替换 `RuleLoader` 的字典输出。
2. **定义协议**：在 `engine` 目录下新建 `models.py`，定义统一的 `Issue` 和 `ParagraphNode` 类。
3. **肢解 Validator**：将 `Validator.validate()` 拆分成多个小的 `Plugin` 类，用一个列表统一管理。
4. **切断 UI 耦合**：确保 `src/engine/` 下的所有代码都不包含任何处理网络请求、JSON 序列化或 UI 更新的库，全部改为通过 `yield` 返回状态流。
