# 方案设计：规则管理页面重构 (Rules UI Redesign)

## 1. 现状痛点
- **功能堆叠**: 目前所有规则均在一个长列表中展示，随着规则项增多，由于搜索、导航成本急剧增加。
- **扩展性弱**: 规则配置项硬编码在模板中，未来新增规则（如：分页控制、特殊题注、特定学科校验）会持续拉长页面。
- **主功能不突出**: 高级功能（如“禁用层级检查”）被淹没在基础格式配置中。

## 2. 改进方案 (Proposal)

### 2.1 动态分类导向
引入 **二级子标签页 (Sub-tabs)** 或 **Segmented Control** 模式，将规则配置划分为：
1. **📁 常规 (General)**
   - 全局默认设置 (`document`)
   - 页面布局设置 (`page_setup`)
   - 规则库切换与热重载
2. **🏷️ 标题 (Headings)**
   - 一级、二级、三级标题的格式定义。
3. **📄 正文/备注 (Content)**
   - 正文段落模型、图表注、参考文献。
4. **🛠️ 扩展/插件 (Advanced)**
   - 分页控制插件 (`PaginationPlugin`) 设置。
   - 高级验证器 (`Validators`) 开关（层级审查、GB/T 格式）。

### 2.2 视觉与交互 (UX/UI)
- **卡片化呈现 (Glassmorphism)**: 延续系统深色玻璃风格。
- **Master Toggle (主开关)**: 为高级插件（如分页控制）提供显眼的模块开关，点击即触发状态联动。
- **平滑切换**: 切换 Tab 时具有微妙的位移过渡动画。

### 2.3 技术实施细节
- **前端状态管理**: 引入 `rulesSubTab` 局部状态。
- **规则配置初始化**: 在进入 Rules Tab 时，动态检查 `fullRules` 的各个节点，确保在绑定 `v-model` 前响应式数据已结构化完成。
- **后端同步**: 保存逻辑保持原有 `save_json` 接口，仅在前端进行逻辑切片。
